# White-labeling a Pyrite deployment

Pyrite ships as "Pyrite" out of the box, but an operator running a
branded instance (e.g. `investigate.transparencycascade.org`) can
re-skin the UI and rendered output under a custom identity. This is
**cosmetic re-skinning, not attribution removal** — Pyrite is open
source and the "Powered by Pyrite" credit always remains visible in
the web UI footer and in KB export trailers.

## Quick start

1. Create a branding folder:

   ```
   /etc/pyrite/branding/
   ├── branding.yaml
   ├── logo.png          # optional, 512×512
   └── wordmark.png      # optional, wide banner
   ```

2. Point Pyrite at it:

   ```bash
   export PYRITE_BRANDING_DIR=/etc/pyrite/branding
   pyrite server
   ```

3. The web UI, site-cache export, KB README footer, and MCP
   research-topic system prompt all render under the new identity.

Unsetting `PYRITE_BRANDING_DIR` restores defaults — no code changes
needed.

See `deploy/branding-examples/transparency-cascade/` for a complete,
production-ready example with real assets and an annotated
`branding.yaml`.

## The branding folder

```
branding/
├── branding.yaml          # canonical config (required to brand)
├── logo.png               # square icon (sidebar, favicon source)
├── wordmark.png           # horizontal wordmark (login / home)
├── favicon.ico            # optional; derived from logo if absent
├── apple-touch-icon.png   # optional; 180×180
└── og-image.png           # optional; 1200×630 social-share card
```

All files except `branding.yaml` are optional. If an asset is not
shipped, the corresponding surface falls back gracefully to a text
rendering of the brand name.

## `branding.yaml` fields

| Field | Purpose |
|-------|---------|
| `name` | Display name. Replaces "Pyrite" in the sidebar, page titles, login/register headings, site-cache titles, JSON-LD publisher, KB export footer. |
| `tagline` | Optional subtitle. Surfaced on the landing page. |
| `primary_color` | Hex color. Drives the `--brand-primary` CSS custom property used for accent coloring. |
| `site_url` | Canonical URL for `<meta>` / `og:url`. |
| `support_url` | Optional support/contact URL shown in the UI. Accepts `mailto:` URIs. |
| `footer_credit_url` | Where the "Powered by Pyrite" footer link points. Default: `https://pyrite.wiki`. |
| `logo` | Filename of the square icon (relative to the branding folder). |
| `wordmark` | Filename of the horizontal wordmark. |
| `favicon` | Filename of the browser-tab favicon. |
| `apple_touch_icon` | Filename of the iOS home-screen icon. |
| `invert_on_dark` | When `true`, the UI applies `filter: invert(1)` to logo/wordmark on dark theme. Ship a single black-on-transparent asset and let CSS invert it for dark mode. |
| `meta.description` | Page-level meta description. |
| `meta.og_image_path` | Filename of the social-share card. |
| `mcp.agent_prompt_brand` | Brand name injected into MCP research-topic system prompts. Defaults to `name`. The MCP protocol handshake itself still announces Pyrite. |

### Theme variants (optional)

If CSS inversion doesn't give you the color you need — e.g. a brand
that enforces specific hex values that aren't pure black or white —
ship explicit light/dark variants:

```yaml
logo_light: "logo-black.png"
logo_dark: "logo-white.png"
wordmark_light: "wordmark-black.png"
wordmark_dark: "wordmark-white.png"
invert_on_dark: false  # disable CSS inversion when variants are set
```

The frontend picks the variant matching the active theme.

## What gets branded

| Surface | Branded? |
|---------|----------|
| Web UI sidebar, login/register/home headings | Yes — rendered from the store |
| Browser tab title | Yes — updated on hydration (brief "Pyrite" flash before) |
| Site-cache HTML (`<title>`, og tags, JSON-LD publisher) | Yes |
| KB export (README) footer | Yes — "Generated by {brand} (powered by Pyrite) on DATE" |
| MCP `research_topic` system prompt | Yes — "access to {brand} knowledge bases (powered by Pyrite)" |
| `<footer>Powered by Pyrite</footer>` | **Always visible** — cannot be disabled |
| HTTP User-Agents (clipper, URL checker) | **Stays `Pyrite-*`** — honest on-the-wire identity |
| MCP protocol handshake software name | **Stays `Pyrite`** — machine identity |
| CLI (`pyrite …`), Python package name, log prefixes | **Unchanged** |

## Operational notes

- **Site-cache rebuild.** When you change `branding.yaml`, the
  already-rendered HTML in `site-cache/` is stale until regenerated.
  Run `pyrite index rebuild-site` (or the equivalent re-seed step in
  your deploy pipeline) after branding edits.
- **CDN caching.** `/config/branding` and `/branding/{file}` are
  public endpoints with no explicit cache headers in V1. Behind a CDN,
  assume these may be cached for a few minutes after edits.
- **Asset path safety.** The `/branding/{filename}` endpoint rejects
  any path containing `..` or absolute-path components. Only files
  directly inside the configured branding folder are served.
- **Env-var precedence.** `PYRITE_BRANDING_DIR` is read at startup
  only. Restart the Pyrite server after changing the env var.

## Out of scope (V1)

- Full theme palette customization beyond a single `primary_color`
- Per-KB branding (multi-tenant use case)
- Runtime admin UI for editing branding
- Email template branding (no emails sent today)

If you need any of these, file a ticket against the
`epic-pyrite-publication-strategy` epic.
