## Problem

Pyrite's UI hardcodes the name "Pyrite" in ~6 visible locations across
the web frontend plus several in server-rendered export HTML. To deploy
a branded instance (first target:
`investigate.transparencycascade.org`), an operator needs to replace
those strings without forking the code.

White-labeling here is **cosmetic re-skinning, not identity
obfuscation**. Pyrite is open source; operators should be free and
encouraged to credit it. The goal is letting a deploy feel like a
product belonging to the investigation (Transparency Cascade, a
specific newsroom, etc.) while keeping "Powered by Pyrite" visible.

## Scope — what gets branded

### Branded (user-facing chrome)

- Sidebar title (`web/src/lib/components/layout/Sidebar.svelte:117`)
- Login / register / home page headings
  (`web/src/routes/login/+page.svelte:34,39`,
  `web/src/routes/register/+page.svelte:48,53`,
  `web/src/routes/+page.svelte:19,30`)
- Browser tab title (`web/src/app.html:7` + per-route `<svelte:head>`)
- Favicon (`web/static/favicon.svg`)
- Site-cache rendered HTML titles and JSON-LD `publisher`
  (`pyrite/services/site_cache.py:221,223,272,274,339,388,571`)
- Site-cache entry-footer "Edit on Pyrite" link text
  (`pyrite/services/site_cache.py:556`)
- KB export footer text "*Generated by Pyrite…*"
  (`pyrite/services/kb_service.py:879`)
- Primary accent color (currently hardcoded `gold-400` — configurable
  CSS custom property driven by the brand config)
- MCP agent system prompt: "access to Pyrite knowledge bases"
  (`pyrite/server/mcp_server.py:1373`) becomes "access to {BRAND}
  knowledge bases (powered by Pyrite)"

### Not branded (machine identity stays Pyrite)

- HTTP User-Agents for clipper and URL-checker
  (`pyrite/services/clipper.py:103`,
  `pyrite/services/url_checker.py:118`) — honest on-the-wire identity
- MCP server software name — still announces itself as Pyrite; brand
  appears in the agent system prompt, not the protocol handshake
- Python package name, CLI command `pyrite`, log prefixes

### Always present (credit to Pyrite)

- Footer line on web UI: "Powered by [Pyrite](https://…)" — not
  configurable off; the brand config only sets what appears *above*
  it
- KB export footer keeps "Generated by Pyrite" as the trailing credit
  after any brand-specific text

## Configuration mechanism — branding folder

A filesystem folder containing the brand config and assets.
Operator points Pyrite at it via one env var:
`PYRITE_BRANDING_DIR=/etc/pyrite/branding`.

Expected layout:

```
branding/
├── branding.yaml          # canonical config
├── logo.png               # square icon (sidebar, favicon source)
├── wordmark.png           # horizontal wordmark (login, home, large headings) — optional
├── favicon.ico            # optional; derived from logo if absent
├── apple-touch-icon.png   # optional; 180×180
└── og-image.png           # optional; 1200×630 social-share card
```

Assets can be **one tone** (black-on-transparent or white-on-transparent).
The `invert_on_dark` flag tells the frontend whether to apply CSS
`filter: invert(1)` on dark theme so one asset serves both themes.
Operators with brand-enforced specific color values can ship both
variants — see "Theme variants" below.

`branding.yaml` shape:

```yaml
name: "Transparency Cascade Press"
tagline: "Tracing the money, naming the actors, connecting the patterns"  # optional
primary_color: "#c93b3b"                                # hex; drives CSS custom prop
site_url: "https://investigate.transparencycascade.org" # canonical URL for og/meta
support_url: "mailto:contact@transparencycascade.org"   # optional
footer_credit_url: "https://pyrite.wiki"                # where "Powered by Pyrite" links

logo: "logo.png"              # square icon, relative to branding dir
wordmark: "wordmark.png"      # optional horizontal wordmark
invert_on_dark: true          # apply CSS filter: invert(1) on dark theme

meta:
  description: "Independent investigative journalism documenting institutional capture, corruption, and threats to democratic accountability."
  og_image_path: "og-image.png"  # optional; relative to branding dir

mcp:
  agent_prompt_brand: "Transparency Cascade"  # optional; defaults to `name`
```

### Theme variants (optional, for operators who need them)

If CSS inversion is insufficient (brand enforces specific hex values
that aren't pure black/white), operators can ship both variants:

```yaml
logo: "logo.png"              # default, used when theme-specific unset
logo_light: "logo-black.png"  # used on light backgrounds
logo_dark: "logo-white.png"   # used on dark backgrounds
wordmark_light: "wordmark-black.png"
wordmark_dark: "wordmark-white.png"
invert_on_dark: false         # disable inversion when variants are set
```

Frontend picks the variant matching the active theme; falls back to
`logo` / `wordmark` if no variant is defined.

Defaults: if `PYRITE_BRANDING_DIR` is unset or the folder is empty,
Pyrite falls back to its built-in branding (name: "Pyrite", default
favicon, default color). No behavior change for anyone not
white-labeling.

Asset resolution: files in the branding dir are served by a new
`GET /branding/{filename}` endpoint. The frontend references them via
the config endpoint (next section) — no hardcoded filesystem paths
leak to the client.

## Runtime endpoint

`GET /api/config/branding` returns:

```json
{
  "name": "Transparency Cascade Press",
  "tagline": "Tracing the money, naming the actors, connecting the patterns",
  "primary_color": "#c93b3b",
  "logo_url": "/branding/logo.png",
  "wordmark_url": "/branding/wordmark.png",
  "invert_on_dark": true,
  "favicon_url": "/branding/favicon.ico",
  "og_image_url": "/branding/og-image.png",
  "support_url": "mailto:contact@transparencycascade.org",
  "footer_credit_url": "https://pyrite.wiki",
  "meta": {"description": "Independent investigative journalism documenting institutional capture, corruption, and threats to democratic accountability."},
  "site_url": "https://investigate.transparencycascade.org"
}
```

`logo_url` / `wordmark_url` may be `null` if the asset file is absent
— frontend falls back to text rendering of `name`. `favicon_url` is
also nullable; if absent and a logo exists, server can synthesize one
(optional enhancement — V1 can just return the default Pyrite favicon
when none is set).

Public endpoint (no auth required — the login page itself needs it).
Response is cacheable; add an ETag based on the branding dir's mtime
so clients can re-check cheaply.

## Frontend wiring

- New `$lib/stores/brand.svelte.ts` store, loaded alongside
  `authStore.init()` in `+layout.svelte`
- Replace 6 hardcoded `"Pyrite"` strings with `{brandStore.name}`
- Root layout sets CSS custom property `--brand-primary` from
  `primary_color`; Tailwind's `gold-400` references in the 3 relevant
  components become `var(--brand-primary)`
- `<svelte:head>` in each top-level route sets the title from
  `brandStore.name`
- Sidebar logo: if `brandStore.logo_url` is set, render it; else
  render the text title. Apply `filter: invert(1)` when
  `brandStore.invert_on_dark` is true and the active theme is dark.
- Login / home: if `brandStore.wordmark_url` is set, render the
  wordmark above the H1 (hiding or demoting the text H1); else keep
  the text H1
- Footer component (new, small): renders
  "Powered by [Pyrite]({brandStore.footer_credit_url})" — always
  visible in the sidebar or page footer

`app.html` can't access the store (it's static HTML), so the initial
`<title>` stays "Pyrite" and SvelteKit's hydration updates it on mount.
Acceptable — the flash is <100ms.

## Server-side wiring

- `pyrite/services/branding_service.py` (new) — loads
  `branding.yaml`, validates, caches, exposes `BrandingConfig` dataclass
- `pyrite/server/endpoints/branding.py` (new) — the two endpoints
  above
- `site_cache.py` accepts a `BrandingConfig` parameter (or reads from
  the service) and uses it for rendered titles, JSON-LD publisher,
  entry-footer link text; keeps "powered by Pyrite" in the rendered
  page footer
- `kb_service.py:879` export footer becomes
  f"*Generated by {brand.name} (powered by Pyrite) on {date}*"
- `mcp_server.py:1373` system prompt reads from
  `BrandingConfig.mcp_agent_prompt_brand`

## Done when

- `PYRITE_BRANDING_DIR=/path/to/branding` with a custom
  `branding.yaml` + logo renders the specified brand name in sidebar,
  login, register, home, browser tab, and site-cache exports
- "Powered by Pyrite" link remains visible in the footer and export
  trailer
- MCP agent's system prompt references the brand name
- Unsetting `PYRITE_BRANDING_DIR` restores "Pyrite" default with zero
  code changes
- `pytest tests/ -v` green
- `web && npm run build` succeeds

## Explicitly out of scope

- Full theme palette customization (only a single `primary_color`
  token in V1)
- Runtime admin UI for editing branding (file + env var is enough for
  V1 — add admin UI later if multiple operators request it)
- Per-KB branding (multi-tenant — future ticket)
- PWA manifest + full social-share metadata — pairs naturally but
  tracked separately
- Email templates — no emails are sent today; brand-aware email
  templates land with the first email-sending feature
